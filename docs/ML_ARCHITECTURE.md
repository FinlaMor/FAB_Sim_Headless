# FAB_Sim_Headless — ML Architecture & Data Flow (Technical Reference)

*Last updated: 2026-06-24. Companion to `docs/HOWTO_RUN.md` (operations). This document explains
**what is being learned, with what methods, how the parts connect, and exactly what data flows in and
out of each one.***

---

## 1. The problem and the overall strategy

We want bots that play **Classic Constructed (CC) Flesh and Blood** well, across three decisions:

1. **In-game play** — given a board state and the legal moves, which move? *(the gameplay model)*
2. **Sideboarding** — given my hero + the opponent's hero, which 60 cards + equipment to run? *(the
   sideboard model)*
3. **Deckbuilding** *(future, "Stage 3")* — construct a deck from the entire legal card pool by
   winrate. Not built yet; see §9.

The unifying philosophy: **the PHP Talishar engine is the only source of game rules.** Python never
re-implements rules — it observes the legal action set the engine offers and chooses among them.
Every model is trained **offline** on data generated by self-play through that engine.

Two distinct learning problems, two data streams, two model families:

| Decision | Method | Model file | Trained from |
|---|---|---|---|
| In-game play | **IQL** (offline RL) | `outputs/models/cc_warm*/iql_gameplay.pt` | `datasets/cc/parquet/games/*.parquet` |
| Sideboarding (prior) | **Behavior cloning** | `outputs/models/sideboard/sideboard_bc.pt` | `decks/cc_*.json` (human choices) |
| Sideboarding (refine) | **AWR** (winrate bandit) | `outputs/models/sideboard/sideboard_rl.pt` | `outputs/cc_sideboard_matches.jsonl` |

## 2. The data-generation loop (where everything comes from)

```
                         ┌─────────────────────────────────────────────┐
                         │  cc_selfplay.py  (the driver, one per run)    │
                         │                                               │
 sideboard model ──pick──►  build each side's legal deck (sideboard.    │
 (BC or RL, or human       │   resolve → 60 cards + equipment)           │
  matchup data)            │                                            │
                           │   for each game:                            │
 gameplay model ──play──►  │     loop: GET legal actions → choose →      │
 (cc_warm3)                │           POST /step  (engine applies)      │
                           │     until winner or step_cap                │
                           └───────────────┬────────────────┬───────────┘
                                           │                │
              one row per in-game decision │                │ one row per matchup
                                           ▼                ▼
              datasets/cc/parquet/games/*.parquet     outputs/cc_sideboard_matches.jsonl
                      (gameplay training)                 (Stage-2 sideboard training)
```

- Games are **sharded across the 8 adapter workers** (a `ThreadPoolExecutor`, one `TalisharEnv` HTTP
  client per worker).
- The engine is **stateless per request**: each `/step` reloads `gamestate.txt`, applies the chosen
  action via the real `ProcessInput()`, and writes the new state back. State lives on disk in
  `datasets/games/{id}/` (one directory per game).
- **Determinism:** a given `--base-seed` reproduces the same matchups, decks, and RNG, so runs are
  replayable and A/B-comparable.

## 3. The gameplay model (IQL) — the core of the project

### 3.1 What it is

An **Implicit Q-Learning** policy over the game's per-decision transitions. IQL is offline RL: it
learns a value function `V`, an action-value `Q`, and a policy `π` purely from logged trajectories,
**without ever querying the environment during training** (which matters because our environment is a
slow PHP engine). The policy is extracted by **advantage-weighted regression (AWR)**: imitate the
logged actions, but weight each by how much better than average it was.

Code: `python/training/iql_gameplay.py`. Inference bot: `python/gameplay/bots/iql_bot.py`.

### 3.2 Network architecture

A small **Transformer encoder** over the game state, with three MLP heads (`V`, `Q`, `π`).
(`build_net`, `IQLNet`; defaults: `emb_dim=64`, `hidden=256`, `n_heads=4`, `n_layers=2`.)

**State encoding** — the state becomes a sequence of tokens:
```
[ CTX ,  my-hero ,  opp-hero ,  hand-card-1 … hand-card-K ,  (other visible zone slots) ]
```
- Each **card token** = `card-id embedding` + `role embedding` (which zone) + `attribute projection`
  (the card's features from the attribute table) + `per-token state projection` (counters,
  ready/face-up flags).
- The **CTX token** carries the state *scalars* (phase, both life totals, combat-awareness signals),
  projected to embedding width.
- Self-attention runs over all tokens (`src_key_padding_mask` hides empty/`PAD` slots and masked
  opponent cards). **The CTX token's output is the state vector `sv`.** Attention (rather than
  mean-pooling the hand) lets the policy reason about hand/board synergies.

**Action encoding** — each legal action = `action scalars` (its type/cost/target features) +
`card-id embedding` of the card it plays/uses. Non-card actions (PASS, decisions) just have a `PAD`
card. This makes the variable-length, heterogeneous action set trivial to score.

**Heads** (each a 3-layer MLP):
- `V(sv)` → scalar state value.
- `Q(sv, av)` → scalar action value.
- `π(sv, av)` → scalar logit per legal action; softmax over the legal set = the policy.

This design means **the action space never has a fixed head** — actions are scored against the state
vector, so any number/kind of legal actions works.

### 3.3 Inputs & outputs

| | |
|---|---|
| **Train input** | `datasets/cc/parquet/games/*.parquet` (transition rows, §6.1) + `cc_card_table.pt` |
| **Train output** | `outputs/models/cc_warmN/iql_gameplay.pt` (net weights + vocab + `attr_dim` + hypers + metadata) |
| **Inference input** | live state JSON + legal action list from the adapter |
| **Inference output** | one chosen `action_id` (argmax of `π`, or temperature-sampled) |

### 3.4 Reward & objective

Reward is **terminal and zero-sum** from the mover's perspective: `+1` win, `−1` loss. Several
*shaping* terms (all configurable hypers) make the sparse signal learnable:

- **Shaped terminal** (`use_shaped_reward`, on): scale the ±1 by the life margin (a 40-life blowout
  is worth more than a 1-life squeak).
- **`draw_penalty`** (0.3): a step-cap draw's final recorded decision becomes a terminal with reward
  `−0.3`, so the value ordering is **win (+0.5..1) > draw (−0.3) > loss (−1)**. This is the
  non-potential lever that actually pushes the policy to *close games* instead of coasting to the
  clock. (Without it, ~92% of games drew — a draw at reward 0 beats risking a −1 loss.)
- **`time_penalty`** (0.002): tiny per-decision living cost on all games → prefer faster wins.
- **`aggression_weight`** (0, optional): potential-based tempo shaping on life-lead change.

**Losses** (the IQL recipe + two project-specific additions):
- **Expectile value loss** (`expectile_tau=0.7`): `V` regresses toward the upper expectile of `Q`,
  the IQL trick that approximates a max without querying out-of-distribution actions.
- **Q Bellman backup** with a slow target network (`target_tau=0.005`), negamax sign convention
  (`sign = +1` if the next mover is the same player, else `−1`).
- **AWR policy loss**: cross-entropy on the logged action, weighted by `exp(beta * advantage)` with
  `advantage = Q − V`. `advantage_beta=3`, advantages **z-scored per batch** (`adv_norm=True`) — the
  audit found raw advantages so tiny that the weights were ~uniform, silently degrading AWR into
  plain behavior cloning; normalizing restores real good-vs-bad contrast.
- **MC-return auxiliary** (`mc_aux_weight=1.0`): regress `V` on the realized negamax Monte-Carlo
  return of each transition, injecting the actual win/loss outcome directly at every state (fights
  value under-dispersion through long ~85-decision games).
- **Terminal upsampling** (`terminal_upsample=8.0`): terminals are ~1/85 of rows, so the ±1 anchors
  are sampled 8× more often or `V` drifts toward predicting ~0 everywhere.

### 3.5 Known weakness (be aware)

The **critic under-disperses**: `V`/`Q` losses sit very low and `corr(V, return)` has historically
been weak (the "critic collapse" finding). `adv_norm`, `mc_aux_weight`, `terminal_upsample`, and the
256/64 capacity bump are all mitigations. The practical consequence: gains per iteration are modest;
the lever that most moves *decisive* play is `draw_penalty` + more/cleaner data, not architecture.
Audit it with `python -m python.examples.analyze_qv`.

### 3.6 Lineage

`cc → cc_warm → cc_warm2 → cc_warm3 (78% decisive, 114K transitions) → cc_warm4 (current champion,
promoted 2026-06-25; warm-started from cc_warm3 on the 400K post-Coercive corpus with the fixed
emb/attr_proj inheritance)`. Each is a warm-start (`--warm-start`) from the previous champion; a new one
only becomes champion if it wins the decisive head-to-head **gate** (§8). Note cc_warm4's gate vs
cc_warm3 was a statistical tie (51.7% ± 5.4%) — it was promoted on the "more data + correct warm-start,
not worse" rationale, not a clear win.

## 4. The card attribute table — giving the model "eyes"

File: `outputs/models/cc/cc_card_table.pt`. Built by `python/training/build_cc_card_table.py` from
`slug_index.json`.

- A **vocabulary** of 4728 tokens = (every CC card slug) ∪ (state sentinels the serializer emits);
  noise tokens are filtered to `UNK`.
- An **attribute matrix** `[n_cards, attr_dim=79]`: per-card features (cost, pitch, power, defense,
  class, types, keywords, …) from `python/deckbuilding/cc_card_attrs.py`.

This matrix is loaded as a **frozen buffer** inside the gameplay net and projected into the card
embedding (`_card(ids) = emb(ids) + attr_proj(attr_table[ids])`). Effect: the model generalizes
across cards that *share attributes* instead of memorizing 4728 independent slugs — crucial for a
4728-card vocabulary with limited data. **Without it**, the model is "card-blind" and only learns
from the scalar state (a tempo prior). This is why `--card-table` is mandatory for CC training, and
why the target net must be built with the same `attr_dim`.

## 5. The sideboard models (deck selection per matchup)

The gameplay model plays a *given* deck. The sideboard model decides *which* deck — specifically, how
many copies (0–3) of each card in the hero's pool to maindeck against a particular opponent.

### 5.1 Stage 1 — Behavior Cloning (`sideboard_model.py`)

**Goal:** imitate how expert deck authors adjust their 60 + sideboard per matchup, so a *novel*
opponent gets a learned plan instead of a crude class-similarity fallback.

- **Input data:** scraped human decks `decks/cc_*.json`, each with a base `deck`/`equipment`, a
  `sideboard`, and `matchup_quantities` (the author's per-opponent card counts).
- **Features per card** (`card_features`, `FEAT_DIM` ≈ 60-d): normalized cost, pitch one-hot,
  power/defense, and multi-hot over 16 classes / 12 types / 15 keywords — all from `slug_index.json`.
  So the model keys on card *properties*, not slugs (generalizes to unseen cards).
- **Net:** `[my-hero embedding ‖ opp-hero embedding ‖ card-feature projection] → MLP → softmax over
  counts {0,1,2,3}`.
- **Training:** weighted cross-entropy; examples where the matchup count **differs from base** (the
  actual sideboard *decisions*) are upweighted (`delta_weight=4`) so they aren't drowned out by the
  many cards that never change. Honest 20% held-out split, reported separately on the delta examples
  (~73% held-out delta-accuracy).
- **Output:** `sideboard_bc.pt`. At inference, `predict_overrides(pool, opp_hero)` → `{slug: count}`,
  which feeds `sideboard.resolve(pool, opp_hero, overrides=…)` to produce a legal deck.
- **Exploration hook:** `sample_overrides(pool, opp_hero, temperature)` samples counts from
  `softmax(logits/T)` instead of argmax — this is what `--explore-sideboard` uses to create the
  choice *variation* Stage 2 needs.

### 5.2 Stage 2 — Winrate RL via AWR (`sideboard_rl.py`)

**Goal:** move beyond imitating authors toward choices that *actually win* in self-play.

- **Input data:** `outputs/cc_sideboard_matches.jsonl` — one row per matchup: which overrides each
  side chose and who won (§6.2). Generated by collection with `--explore-sideboard > 0`.
- **Method:** advantage-weighted regression treating each matchup as a **contextual bandit** (not a
  sequential RL problem — the "action" is the whole sideboard choice, the "reward" is the game
  outcome). Warm-started from the BC checkpoint. `π_loss = mean(exp(beta * reward) * CE)` plus a
  **KL anchor to the frozen BC** (`bc_anchor=0.5`) so thin data can't push it into nonsense.
- **Guards:** `min_decisive=20` (won't train on too few decisive games); only decisive games count
  (draws ignored, ties never broken by life).
- **Output:** `outputs/models/sideboard/sideboard_rl.pt` (same architecture/interface as BC, so it
  drops into `--model` identically).
- **Status:** pipeline built and validated end-to-end on synthetic data; waiting on a large enough
  decisive corpus from collection.

### 5.3 `sideboard.resolve()` — the legality backstop

`python/deckbuilding/sideboard.py::resolve(pool, opp_hero, overrides=…)` turns a pool + per-card
counts into a **legal** deck: applies the overrides, then enforces hard rules with fallbacks —
- if equipment ended up empty → restore the pool's starting equipment;
- if the maindeck is <60 after sideboarding → top up from the base deck;
- if all equipment got sideboarded out → restore from `sideboard_equipment`.

These backstops exist because the models can output illegal combinations; a verified sweep of 6808
combos produced 0 illegal decks. Illegal decks are *rejected*, never played — they don't corrupt data.

## 6. Data schemas (exact fields)

### 6.1 Game transition row — `datasets/cc/parquet/games/*.parquet`

One row per **recorded decision** (forced single-option windows are dropped — they're not decisions
and would swamp the gradient toward PASS). Fields consumed by the trainer:

| Field | Meaning |
|---|---|
| `game_id` | groups rows into a game (for MC returns) |
| `step_index` | ordering within a game |
| `player_to_move` | 1 or 2 — whose decision this row is (the "mover") |
| `state_json` | full visible state at decision time |
| `next_state_json` | state after the chosen action resolved |
| `chosen_action_json` | the action that was taken |
| `legal_actions_json` | the full legal set offered (the choice context) |
| `done` | terminal flag |
| `winner` | 0 (draw/ongoing), 1, or 2 |

From these the trainer derives: state tokens + scalars, action features, reward (with shaping),
negamax sign, and the Monte-Carlo return.

### 6.2 Sideboard match row — `outputs/cc_sideboard_matches.jsonl`

One JSON object per matchup:

| Field | Meaning |
|---|---|
| `heroA`, `heroB` | the two heroes |
| `deckA`, `deckB` | which pool variant each used (`_src` id) |
| `overrideA`, `overrideB` | `{slug: count}` sideboard choice each side made |
| `winA`, `winB`, `draws` | outcome tally over the matchup's games |
| `explore_temp` | the sampling temperature used for the choice |
| `run_id`, `ts` | run segmentation + timestamp |

### 6.3 Scraped pool — `decks/cc_*.json`

`hero`, `deck` (base 60), `equipment`, `sideboard`, `sideboard_equipment`, `matchups` (with opponent
`heroIdentifiers`), and `matchup_quantities` (per-matchup `{slug: count}`). This is the Stage-1
training source and the menu of decks self-play draws from.

## 7. The engine/adapter interface (what Python sees)

The adapter (`adapter/`) exposes a tiny REST API per worker: `POST /new_game`, `GET /state`,
`GET /actions`, `POST /step`, `POST /reset`, `GET /health`. The Python client is
`python/gameplay/env.py::TalisharEnv`.

The most important file for ML behavior is **`adapter/serializers/ActionSerializer.php`** — it decides
*which legal actions the bot is even offered*. If an action isn't enumerated here, the policy can
never learn to take it. Historically this caused real gaps: missing arsenal/INSTANT/equipment-block
actions meant the model was trained on a *corrupted action space* and was blind to whole categories
of play (e.g. it never learned to block with equipment until that branch was added). The lesson:
**the action space is part of the model, not just the engine.**

`StateSerializer.php` builds the state JSON the model encodes. `RngHook` seeds RNG deterministically
per step.

## 8. Evaluation: the decisive gate

A candidate gameplay model replaces the champion **only if it wins more decisive games head-to-head**.
Implemented in `continuous_train.py::_gate`:
- candidate vs champion across a **fixed pool of strong human decks**;
- every deck matchup played **both seat orientations** (so deck/seat bias cancels — only play skill
  is measured);
- plays in rounds until **100 decisive games** (draws excluded, never broken by life totals) — a
  fixed *decisive* target keeps statistical power constant as draw rates vary (~SE 5% at 100);
- promote iff `cand_wins > champ_wins`.

This "decisive wins only" rule is a core project principle: **success = winning, not surviving.**
Life totals are never used to decide outcomes (rewards or gates).

## 9. Stage 3 (future): the deckbuilding bot

The stated next-stage goal: a bot that **tunes whole decklists by winrate over all legal cards**, not
just the author's sideboard. It's the natural sequel to Stage 2 — same winrate signal, but the action
space explodes from "keep/cut among ~20 sideboard cards" to "choose 60 from hundreds." Likely shape:
legality-constrained search/optimization over the hero's legal pool, candidates evaluated by
self-play winrate vs a fixed gauntlet, **warm-started from human decks** (the hard-won lesson from the
draft bot: self-play-from-random sat at chance level), reusing the card-attribute table and the
`resolve()` legality backstops. **Gated on** Stage-2 maturity and on more compute — the combinatorial
action space slams straight into the throughput wall (§10).

## 10. Throughput & infrastructure realities

Self-play is the bottleneck for everything. Measured facts (so we don't re-investigate):

- **The one big win:** set `torch.set_num_threads(1)` in the collection driver. With N worker threads
  each spawning torch's default (= #cores) intra-op threads, the box thrashed; pinning to 1 gave a
  **17× inference speedup** (the pool already provides parallelism). This is baked into `cc_selfplay`.
- **~35 games/hour** on this 6-core box at 8 workers, ~59% CPU (latency-bound, not CPU-bound).
- **Dead ends (all measured):** GPU/DirectML (5× *slower* — dispatch overhead + a transformer op
  falls back to CPU; and our inferences are tiny, single-sample), more workers (16 ≈ 8), 2 processes
  (GIL wasn't dominant), adapter serialization trimming (latency-bound, no gain, and a protocol change
  broke determinism), running the driver inside the Docker network (~2 ms/request, trivial vs the
  ~1.1 s/step overhead), and a shared engine lock (checked — all locks are per-game, not the cause).
- **The real ceiling is Docker-Desktop-on-Windows VM contention** (one Linux VM mediating 8 containers
  on 6 cores). The only real accelerant is **horizontal scale (cloud/more cores)** — the engine
  parallelizes across games, so a 32-core box ≈ 4×.

Per-step cost breakdown (instrumented): `applyAction` (engine `ProcessInput`) ~47%, the two
`ActionSerializer::enumerate` calls ~49% combined, state reads ~4%. The enumerate cost is intrinsic to
offering a correct, filtered legal action set (it calls `IsPlayable` per card) and is *why* the bot
never gets useless/illegal actions.

## 11. Glossary

| Term | Meaning |
|---|---|
| **IQL** | Implicit Q-Learning — offline RL that learns V/Q/π from logged data without env queries. |
| **AWR** | Advantage-Weighted Regression — imitate logged actions weighted by `exp(beta·advantage)`. |
| **BC** | Behavior Cloning — supervised imitation of expert choices. |
| **Expectile** | Asymmetric regression target; IQL uses it to approximate a max without OOD actions. |
| **Wedge** | A state the engine/serializer can't resolve, so the game stalls/draws (poisons data). |
| **Decisive game** | A game with a real winner (lethal/engine), as opposed to a step-cap draw. |
| **Override** | A `{slug: count}` adjustment to a deck for a specific matchup (the sideboard choice). |
| **Pool** | A hero's full card universe: base deck + equipment + sideboard + sideboard equipment. |
| **Champion / candidate** | Current best gameplay model vs a newly trained one being gated against it. |
| **Negamax sign** | Perspective flip (`±1`) used in the Bellman backup / MC return for the 2-player zero-sum game. |
```
